# Discussion Draft

The refreshed first-pass results support a more critical answer to the research question than a simple "machine learning versus statistics" contest. The best RMSE is produced by lagged linear regression, while rolling historical volatility is very close behind. Random Forest remains weaker than both of these simpler models, and GARCH(1,1) performs worst on the current target. This means that the project cannot honestly claim that machine learning has improved Bitcoin volatility forecasting in a decisive way.

One important reason is the choice of target. The project forecasts next-day 30-day realised volatility, and this target is itself a rolling measure. Because volatility is persistent, today's realised volatility is already a strong predictor of tomorrow's realised volatility. This naturally favours persistence-based models. The strong performance of rolling historical volatility therefore tells us something meaningful: a complicated model must clear a high bar before it deserves to be called genuinely better.

The current Random Forest result is still useful even though it does not win. Its feature-importance output shows that the model relies mostly on current realised volatility, 30-day rolling return standard deviation, and recent realised-volatility lags. In other words, even the machine-learning model is mainly learning persistence rather than a completely new nonlinear structure. This supports the interpretability discussion, because it suggests that much of the forecast signal is already visible in simpler lag-based variables.

The GARCH result should also be interpreted carefully rather than dismissed too quickly. GARCH(1,1) forecasts conditional daily variance, whereas the evaluation target is a next-day 30-day realised-volatility proxy. The project converts the one-step-ahead GARCH variance into a comparable rolling-volatility forecast, but this is still a less direct match to the target than simply carrying forward recent realised volatility. This means the poor GARCH ranking may partly reflect target mismatch as well as model weakness.

Another important limitation is scale. The project uses daily Hyperliquid BTC perpetual futures candles rather than intraday returns, order-book data, sentiment measures, or macro variables. Many academic machine-learning papers use richer feature sets or more heavily tuned neural networks. This EPQ therefore tests whether a relatively small and reproducible implementation can justify its added complexity. At the moment, the answer appears to be no.

This does not make the project unsuccessful. In fact, the opposite is true. A strong evaluative project does not need machine learning to win. It needs to explain clearly why the results came out the way they did and what that implies. The current evidence supports the argument that simple, transparent, and reproducible models remain highly competitive for this forecasting target. That is a meaningful conclusion for both methodology and practical risk management.

The key decision now is no longer whether to implement LSTM, because that step has been completed. The more important decision is how to interpret the LSTM result. The most defensible reading is that sequence-based machine learning can add some predictive value relative to the Random Forest, but that the gain is still not large enough to beat the strongest simple alternatives. This allows the final report to stay balanced rather than framing LSTM as either a guaranteed winner or a failed idea.
